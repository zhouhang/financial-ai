## Context

### 背景与现状

当前系统强制用户登录才能使用AI对话和上传对账。新文件进行用户必须先注册/登录才能体验任何功能，导致使用门槛过高。

### 当前架构

- **认证系统**: JWT token (24小时有效期), 存储在 `tally_auth_token`
- **对账规则**: 存储在 PostgreSQL `reconciliation_rules` 表，按 company_id/department_id 隔离
- **用户会话**: 存储在 PostgreSQL `conversations` 表
- **前端UI**: 固定顶部tab栏 + 侧边栏登录状态显示

### 约束条件

1. 游客数据不能持久化到用户账户（规则不对应具体用户）
2. 游客使用必须有限制，防止滥用
3. 登录流程不能有重大变更（兼容现有用户）
4. MCP工具需要支持临时token认证

### 涉及的利益相关者

- 新用户（希望快速体验功能）
- 已注册用户（不受影响）
- 运营团队（需要控制游客使用量）

## Goals / Non-Goals

### Goals

1. 允许用户无需登录即可与AI对话，AI可正常介绍自己和功能
2. 未登录用户可上传文件进行对账，仅使用系统推荐规则
3. 实现临时auth_token机制（MCP工具派发，7天有效）
4. 游客对账功能最多使用3次，超过后提示登录
5. 移除顶部"分析会话"tab栏，未登录用户右上角显示登录按钮
6. 实现页面弹窗登录（替代AI表单登录）
7. 登录完成后自动保存推荐规则并刷新页面

### Non-Goals

- 游客会话持久化（不保存到数据库）
- 游客导出对账结果（仅展示）
- 游客创建/修改规则（仅使用推荐规则）
- 游客历史记录查询

## Decisions

### D1: 临时token存储方案

**决定**: 新建 PostgreSQL 表 `guest_auth_tokens` 存储临时token

**理由**: 
- 现有 `auth_tokens` 表是内存/Redis方案，不适合持久化
- 临时token需要7天有效期，需要持久化存储
- 与现有用户认证体系解耦

**备选方案**:
- [方案B] 复用现有token表 → 排除：游客token不应与用户token混淆
- [方案C] 存储在前端localStorage → 排除：无法限制多设备使用次数

### D2: 游客分支设计

**决定**: 在 LangGraph main_graph 中新增独立的 `guest` 路由分支

**理由**:
- 与现有登录用户流程解耦，避免相互影响
- 可独立控制游客的功能权限
- 便于后续扩展其他游客能力

**备选方案**:
- [方案B] 在现有流程中增加条件判断 → 排除：逻辑复杂度高，容易遗漏
- [方案C] 独立的服务/端点 → 排除：代码重复，维护成本高

### D3: 登录UI方案

**决定**: 使用 React Portal 实现页面弹窗登录

**理由**:
- 不依赖AI对话流程，用户体验更好
- 可复用现有的登录API
- 实现相对简单，无需新增后端接口

**备选方案**:
- [方案B] 新开页面登录 → 排除：需要处理URL路由和状态传递
- [方案C] 侧边栏内嵌表单 → 排除：空间有限，体验不佳

### D4: 使用次数计数方案

**决定**: 在 `guest_auth_tokens` 表中记录 `usage_count` 字段，每次对账成功后+1

**理由**:
- 简单可靠，与token生命周期绑定
- 便于审计和统计

**备选方案**:
- [方案B] 前端localStorage计数 → 排除：可被篡改
- [方案C] IP地址限制 → 排除：多人共用IP时误伤

## Risks / Trade-offs

### R1: 游客数据泄露风险

**风险**: 游客上传的文件/对账结果可能被其他游客看到

**缓解**: 
- 游客数据使用临时session ID隔离
- 对账结果不持久化到数据库，仅内存处理
- 会话结束（浏览器关闭）数据清除

### R2: 3次限制被绕过

**风险**: 用户清除浏览器数据后可重置计数

**缓解**:
- 结合IP限制（同IP每日上限）
- 引导注册降低绕过动机
- 记录设备指纹（可选，增强限制）

### R3: 推荐规则质量

**风险**: 游客无法选择/调整规则，可能匹配不准确

**缓解**:
- 推荐算法使用多维度匹配（字段名、映射哈希、样本数据）
- 登录后可编辑/保存规则
- 持续优化推荐算法

### R4: 弹窗登录用户体验

**风险**: 弹窗被浏览器拦截

**缓解**:
- 首次点击时检测拦截并提示
- 提供fallback: 点击跳转独立登录页

## Migration Plan

### Phase 1: 后端改动

1. 创建 `guest_auth_tokens` 表
2. 新增 MCP 工具: `create_guest_token`, `verify_guest_token`, `increment_guest_usage`
3. 修改现有 MCP 工具支持 `guest_token` 参数

### Phase 2: Data Agent 改动

1. 新增 `guest` 路由分支（判断是否有 `guest_token`）
2. 游客模式下过滤规则列表（仅返回推荐规则）
3. 对账完成后检查使用次数

### Phase 3: 前端改动

1. 移除顶部tab栏
2. 新增登录弹窗组件
3. 修改 Sidebar 登录状态显示
4. 未登录时显示登录按钮
5. 登录成功后触发规则保存和页面刷新

### 验证步骤

1. 测试未登录用户对话流程
2. 测试游客上传文件对账流程
3. 测试3次限制触发
4. 测试弹窗登录
5. 测试登录后规则保存

## Open Questions

### Q1: 游客会话结束后数据保留多久?

当前设计：浏览器关闭后数据清除。需要确认是否需要更长的临时存储？

### Q2: 游客能否查看对账历史?

当前设计：不能。若需要，需重新设计session管理。

### Q3: 登录后是否立即继承游客数据?

当前设计：登录后重新开始。若需要继承游客数据，需要额外设计数据迁移逻辑。
