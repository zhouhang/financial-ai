## Context

当前系统在对账对话中使用"业务文件"和"财务文件"术语，但这是通用对账工具，不应强制区分文件类型。代码中有254处相关引用。

**当前状态**:
- 系统提示词中包含"业务文件和财务文件"描述
- 对话输出使用"业务数据"、"财务数据"、"业务文件"、"财务文件"
- 内部代码使用 business/finance 标识（保持不变）

## Goals / Non-Goals

**Goals:**
- 对话中只使用"文件1/文件2"或通用术语"文件"
- 移除面向用户的业务/财务术语
- 保持内部逻辑不变

**Non-Goals:**
- 不修改数据库字段名
- 不修改 business/finance 内部标识
- 不修改文件类型识别逻辑

## Decisions

### D1: 术语替换策略
**Decision**: 对话输出使用"文件1/文件2"替代"业务文件/财务文件"

**Alternatives considered**:
- "左侧文件/右侧文件" - 需要UI配合，暂不采用
- "A文件/B文件" - 不够直观

**Rationale**: "文件1/文件2"与界面上传顺序一致，用户容易理解

### D2: 系统提示词修改范围
**Decision**: 移除"业务数据"、"财务数据"等面向用户的描述，保留技术提示

**Rationale**: 
- 系统内部仍需识别文件类型进行对比
- 用户看到的是通用术语

### D3: 不修改内部代码
**Decision**: helpers.py 等工具函数中的 business/finance 标识保持不变

**Rationale**: 内部逻辑需要区分两个文件，修改成本高且无用户价值

## Risks / Trade-offs

- [Risk] 已有用户习惯 - 部分用户可能习惯"业务/财务"术语 → [Mitigation] 通过规则配置可自定义显示名称
- [Risk] 配置兼容 - 原有的"业务文件"配置引用需要兼容 → [Mitging] 对用户输入做双向兼容处理
- [Risk] 遗漏位置 - 254处引用可能遗漏 → [Mitigation] 使用 grep 全面检查后进行测试验证
