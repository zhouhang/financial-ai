## Context

当前系统架构：
- 前端通过 WebSocket 与后端通信
- 后端使用 LangGraph 处理 AI 对话逻辑
- AI 回复通过 WebSocket 一次性完整返回
- 前端已有 `stream` 类型消息处理逻辑，但需要完善

现有代码中已部分支持流式输出：
- `App.tsx` 中存在 `stream` 类型消息处理（第130-174行）
- `streamingMessageId` 状态用于追踪流式输出
- 前端已有 `stream` 和 `message` 类型消息的衔接处理

## Goals / Non-Goals

**Goals:**
- 实现 AI 回复的逐字流式输出效果
- 保持与现有消息系统的兼容性
- 提供流畅的用户体验（打字机效果 + 光标动画）
- 确保消息不重复显示

**Non-Goals:**
- 不修改后端 LangGraph 核心逻辑
- 不添加新的 API 端点
- 不支持多模态内容（图片、文件）的流式输出

## Decisions

### 1. 流式输出粒度
**决策**: 使用单词或短句作为流式输出单位，而非逐字符
**理由**: 
- 逐字符推送量太大，网络开销高
- 逐字符渲染性能较差
- 短句（如 2-8 个字符）兼顾流畅度和性能
- 类似 ChatGPT 的用户体验

### 2. 后端推送策略
**决策**: 后端使用 `stream` 类型消息推送，在 LLM 输出完成时发送 `message` 类型完整消息
**理由**:
- 复用现有的 `stream` 消息类型
- `message` 消息作为最终确认，避免内容丢失
- 前端已处理好两种消息的衔接逻辑

### 3. 前端渲染策略
**决策**: 流式输出时在文本末尾添加闪烁光标，提示用户 AI 正在输出
**理由**:
- 光标提供明确的"正在输出"视觉反馈
- 与 ChatGPT 等主流 AI 产品的体验保持一致
- 流式输出完成后光标自动消失

### 4. 消息去重
**决策**: 保留 `stream` 消息累积内容，最终由 `message` 消息覆盖
**理由**:
- 现有代码已有此逻辑（`App.tsx` 第198-209行）
- 简化实现，避免额外状态管理

## Risks / Trade-offs

- **[风险]** 流式输出时用户发送新消息可能导致显示错乱
  - **缓解**: 在 `handleSendMessage` 中清除 `streamingMessageId` 状态，中断当前流式输出

- **[风险]** 网络延迟导致流式输出不连续
  - **缓解**: 前端使用文本追加而非替换，保持显示连续性

- **[风险]** 长文本流式输出性能问题
  - **缓解**: 使用短句而非逐字符推送，控制 DOM 更新频率

- **[风险]** 用户中断流式输出后，后端仍在继续生成
  - **缓解**: 前端发送新消息时通知后端取消当前任务，后端支持任务取消机制
