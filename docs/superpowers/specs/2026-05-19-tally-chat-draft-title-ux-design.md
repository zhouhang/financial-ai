# Tally 对话草稿与任务标题优化设计

日期:2026-05-19

## 问题

Tally 当前把三类对象混在同一个"历史对话"列表里:

- 空的新对话草稿:用户点击"开启新对话"后,还没有发送任何消息。
- 普通聊天会话:用户直接和 Tally 对话。
- 文件任务会话:用户从"上传文件整理"或"上传文件对账"入口选择规则、上传文件、等待 Tally 返回结果文件。

这导致两个明显体验问题:

1. 点击"开启新对话"后,左侧可能出现一个"新对话"历史项;用户点击这个历史项后,它又消失。用户会以为系统丢了会话。
2. 数据整理/数据对账完成后沉淀成无辨识度的"新对话"或用户首句摘要,无法从历史列表判断它是哪次任务、用了哪个规则、是否与文件处理有关。

## 根因

前端目前用 `pendingNewConvRef` 表示"待确认的新会话",但侧边栏展示时会把它合并进历史列表。这个 pending 会话还不是稳定的历史记录,也没有服务端实体,所以它的展示和选择语义不一致。

文件任务入口复用同一套聊天会话模型。选择"上传文件整理/对账"时会创建带 `taskContext` 的临时会话,但标题仍按普通聊天逻辑生成:第一条用户消息截断,或默认"新对话"。任务元信息没有进入历史标题。

## 目标

- 空的新对话草稿不显示在左侧历史列表。
- 用户仍然可以点击"开启新对话"进入空聊天界面,并发送第一条消息。
- 普通聊天只有在发送第一条用户消息后才进入历史列表。
- 数据整理/数据对账会话在进入历史列表时使用任务化标题,便于回看。
- 不改变当前聊天式文件任务流程,不引入任务运行卡,不改后端任务执行链路。

## 非目标

- 不重做数据整理/数据对账为独立任务运行页。
- 不新增数据库字段。
- 不改变会话与消息持久化 API。
- 不改变规则选择、文件上传、WebSocket 消息协议。
- 不重写历史会话加载、删除、登录恢复逻辑。

## 设计

第一阶段采用保守改造:只修正"草稿是否进入历史列表"和"历史标题如何生成"。

### 1. 新对话草稿

"开启新对话"的含义改为:打开一个本地草稿工作区。它不是历史记录。

行为规则:

- 点击"开启新对话"后,主区域显示空聊天界面。
- 左侧历史列表不新增"新对话"条目。
- 如果用户未发送任何消息就点击其它历史会话,该草稿直接丢弃。
- 如果用户在草稿中发送第一条消息,前端创建本地会话并加入历史列表。
- 如果后端随后返回 `conversation_created`,本地 ID 替换为服务端 ID,沿用现有映射逻辑。

这保留了当前 pending 会话的低成本实现方式,但不把 pending 对象展示成历史项。

### 2. 普通聊天标题

普通聊天继续使用首条用户消息生成标题:

```text
<用户首句前 20 个字符>
```

如果用户首句为空或只有附件,使用兜底标题:

```text
新对话
```

这个兜底只用于已实际产生消息的会话,不用于空草稿展示。

### 3. 文件任务标题

当会话存在 `taskContext` 时,标题按任务类型生成,覆盖普通首句摘要。

数据对账:

```text
数据对账 · <规则名> · <YYYY-MM-DD>
```

数据整理:

```text
数据整理 · <规则名> · <文件数>个文件
```

标题字段来源:

- 任务类型:`taskContext.task_type`,识别 `recon` 与 `proc`。
- 规则名:`taskContext.name`,为空时回退 `taskContext.task_name`,再回退"当前规则"。
- 日期:会话创建时间或首次发送时间,格式为本地日期 `YYYY-MM-DD`。
- 文件数:第一条用户消息的 `attachments.length`;若没有附件,显示日期而不是"0个文件"。

数据整理文件数兜底:

```text
数据整理 · <规则名> · <YYYY-MM-DD>
```

数据对账优先展示日期,因为对账常见为多文件或数据源执行,日期比文件数更稳定。

### 4. 标题更新时机

标题应在两个时机计算:

1. 用户首次发送消息时,本地创建会话前计算标题。
2. 文件上传或任务上下文已存在但标题仍是"新对话"时,收到首个服务端消息或 `conversation_created` 后补算标题。

原则:

- 已经拥有任务化标题的会话不再被后续普通消息覆盖。
- 服务端历史加载回来后,若服务端标题仍是"新对话"且本地可判断任务上下文,允许本地展示层使用任务化标题。
- 不在第一阶段做批量修复历史旧标题。

### 5. 侧边栏展示规则

历史列表只展示稳定会话:

- 服务端会话。
- 本地已发送过消息的会话。
- 游客模式中已产生消息的本地会话。

历史列表不展示:

- `pendingNewConvRef` 中 messages 为空的普通新对话草稿。
- 仅因点击"上传文件整理/对账"而创建、但尚未上传/发送的任务草稿。

若当前处于空草稿,侧边栏不需要额外选中一个历史项。主区域标题可显示"Tally 智能财务助手"或"新对话",但这不是历史项。

### 6. 任务上下文清理

点击历史会话时,继续清除临时任务上下文,避免老会话顶部残留当前规则选择器。

但标题不应因为清除 `taskContext` 而被改回普通标题。历史标题是这次会话的档案名,不等同于当前编辑上下文。

## 数据流

### 新建普通草稿

1. 用户点击"开启新对话"。
2. 前端创建 pending 会话,设置为 active。
3. 历史列表不展示该 pending 会话。
4. 用户发送第一条消息。
5. 前端把 pending 转为本地会话,生成普通标题,加入历史列表。
6. 后端创建服务端会话后,现有 `conversation_created` 流程替换 ID。

### 新建文件任务会话

1. 用户点击"上传文件整理"或"上传文件对账"。
2. 前端创建带 `taskContext` 的 pending 会话,设置为 active。
3. 历史列表不展示该 pending 会话。
4. 用户上传文件并发送处理请求。
5. 前端把 pending 转为本地会话,根据 `taskContext` 和附件生成任务标题,加入历史列表。
6. 后端返回结果消息,会话保持任务标题。

## 错误与边界情况

- 用户点击新对话后没有输入就切走:草稿丢弃,不提示。
- 用户点击任务入口后没有上传就切走:任务草稿丢弃,不提示。
- 用户选择任务入口后改成纯文字追问:仍按任务标题生成,因为这是用户主动选择的任务上下文。
- 附件上传失败:不创建历史项;若已创建本地会话但发送失败,保留现有错误处理,不新增回滚逻辑。
- 规则名缺失:标题使用任务名或"当前规则"兜底。
- 服务端旧会话没有任务元数据:保持服务端返回标题,不猜测。

## 测试

前端测试覆盖:

1. 点击"开启新对话"后,主区域为空聊天界面,左侧历史列表不新增"新对话"。
2. 空草稿切换到已有历史会话后,草稿不出现在历史列表。
3. 普通草稿发送首条消息后,历史列表新增该会话,标题为首句摘要。
4. 数据对账入口上传并发送后,历史标题为 `数据对账 · <规则名> · <YYYY-MM-DD>`。
5. 数据整理入口上传两个文件并发送后,历史标题为 `数据整理 · <规则名> · 2个文件`。
6. 后续助手消息到达后,任务标题不被覆盖。
7. 点击历史任务会话后,规则上下文不残留在输入区或顶部规则选择器。

手工验证:

- 登录用户有历史会话时,点击"开启新对话"不会污染历史列表。
- 登录用户刷新页面后,仍按既有逻辑打开最近或已保存的历史会话。
- 游客模式仍可使用一个本地会话。

## 风险

- 当前 `App.tsx` 会话状态较集中,实现时需要避免影响登录恢复、服务端 ID 映射和游客模式。
- 任务标题依赖 `taskContext` 和附件信息,如果某些路径是无附件或数据源执行,必须走日期兜底。
- 历史旧数据不批量修复,用户仍可能看到过去的"新对话"标题;这是第一阶段可接受的范围限制。

## 后续阶段

若第一阶段后仍觉得文件任务流程不够清晰,再单独设计任务运行卡:

- 上传区、规则摘要、执行进度、结果下载、重新运行。
- 聊天作为结果追问入口,而不是承载整个任务运行流程。

该后续阶段需要独立设计和实现计划,不纳入本次保守优化。
